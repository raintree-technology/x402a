===========================================
BUILDING CLAUDE CODE SKILLS
For Aptos Move Smart Contract Development
===========================================

This guide explains how to use the collected documentation to build
effective Claude Code skills for Aptos and Move development.

===========================================
OVERVIEW
===========================================

Claude Code skills allow you to extend Claude's capabilities with
domain-specific knowledge. For Aptos/Move development, you'll want
skills that can:

1. Generate Move smart contracts following best practices
2. Debug and explain existing Move code
3. Suggest security improvements and optimizations
4. Create test suites for Move modules
5. Guide through Aptos-specific patterns (Objects, Fungible Assets, etc.)
6. Help with TypeScript SDK integration

===========================================

1. DEFINE SCOPE
   ------------
   - What specific problem does this skill solve?
   - What Move patterns does it need to know?
   - What are the common use cases?

2. GATHER KNOWLEDGE
   ----------------
   From this documentation collection:

   a. Core concepts:
      - Start with aptos-dev-llms-full.txt
      - Read relevant sections thoroughly
      - Note API signatures and patterns

   b. Practical examples:
      - Find similar examples in move-by-examples/
      - Study their structure and patterns
      - Note how they handle edge cases

   c. Framework reference:
      - Check aptos-core/aptos-move/framework/ for canonical code
      - Understand underlying implementations
      - Note security considerations

   d. Quick reference:
      - Use MOVE_QUICK_REFERENCE.txt for syntax
      - Reference common patterns
      - Check gotchas and best practices

3. CREATE SKILL PROMPT
   -------------------
   Structure your skill prompt with:

   a. Context: "You are an expert in [domain]"

   b. Knowledge base: "You have access to:
      - Aptos official documentation
      - Move language specifications
      - Aptos framework source code
      - Practical examples and patterns"

   c. Capabilities: "You can:
      - [List specific capabilities]
      - [With concrete examples]"

   d. Guidelines: "When generating code:
      - Follow Aptos best practices
      - Include error handling
      - Add descriptive comments
      - Generate tests
      - Consider security implications"

   e. Output format: "Provide:
      - Complete, runnable code
      - Explanations of key decisions
      - Usage examples
      - Testing recommendations"

4. TEST THE SKILL
   --------------
   - Try common use cases
   - Test edge cases
   - Verify code compiles
   - Run generated tests
   - Compare with examples in move-by-examples/

5. ITERATE
   -------
   - Add missing patterns
   - Improve error handling
   - Enhance explanations
   - Update with new Aptos features

===========================================
EXAMPLE SKILL IMPLEMENTATION
===========================================

Here's a sample skill for generating Fungible Asset tokens:

---
Skill Name: "aptos-fa-generator"

Prompt:
"""
You are an expert in Aptos Fungible Asset (FA) development. You have
comprehensive knowledge of the Aptos FA standard, which is the modern
token standard replacing the legacy Coin module.

KNOWLEDGE BASE:
- Aptos Fungible Asset framework (aptos-framework::fungible_asset)
- Primary fungible store pattern
- Object-based metadata
- Ref-based operations (MintRef, BurnRef, TransferRef)

CORE CONCEPTS:
1. Metadata Object: Stores token info (name, symbol, decimals, URIs)
2. FungibleStore: Holds actual token balances
3. Refs: Capabilities for privileged operations
4. Primary Store: Default store for user addresses

STANDARD PATTERN:
```move
module addr::my_token {
    use aptos_framework::fungible_asset::{Self, Metadata};
    use aptos_framework::object::{Self, Object};
    use aptos_framework::primary_fungible_store;
    use std::option;
    use std::string;

    struct MyToken has key {
        mint_ref: fungible_asset::MintRef,
        burn_ref: fungible_asset::BurnRef,
        transfer_ref: fungible_asset::TransferRef,
    }

    public entry fun initialize(creator: &signer) {
        // Create metadata object
        let constructor_ref = &object::create_named_object(
            creator,
            b"MyToken"
        );

        // Initialize FA
        primary_fungible_store::create_primary_store_enabled_fungible_asset(
            constructor_ref,
            option::none(), // unlimited supply
            string::utf8(b"My Token"),
            string::utf8(b"MTK"),
            8, // decimals
            string::utf8(b"https://icon.url"),
            string::utf8(b"https://project.url"),
        );

        // Store refs
        let mint_ref = fungible_asset::generate_mint_ref(constructor_ref);
        let burn_ref = fungible_asset::generate_burn_ref(constructor_ref);
        let transfer_ref = fungible_asset::generate_transfer_ref(constructor_ref);

        move_to(creator, MyToken {
            mint_ref,
            burn_ref,
            transfer_ref,
        });
    }
}
```

When generating FA tokens:
1. Use Object-based metadata (not legacy Coin)
2. Enable primary stores for user convenience
3. Store refs securely (use key ability)
4. Emit events for indexing
5. Add view functions for queries
6. Include comprehensive tests
7. Consider supply limits
8. Handle decimals appropriately

OUTPUT FORMAT:
- Complete module with all necessary imports
- Initialization function
- Mint/burn functions (if applicable)
- Transfer helpers (if needed)
- View functions for balance/metadata
- Events for state changes
- Unit tests
- Deployment instructions
"""
---
